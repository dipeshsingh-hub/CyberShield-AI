# CyberShield AI — Explainable AI-Powered Cybersecurity Threat Detection

SOC analysts don't usually miss a real attack because no tool saw it — they
miss it because ten tools each saw a fragment of it, on separate dashboards,
with no shared story connecting them. A network IDS flags unusual traffic
from one IP. An email gateway flags a suspicious login three minutes later
from a different system. Nothing tells the analyst these are the same
incident, so it either gets triaged as two low-priority alerts or missed
entirely in the noise — the classic alert-fatigue trap, where real signal
and false positives arrive at the same volume and get the same amount of
attention.

CyberShield AI fuses network-behavioral and content-based signals through
genuine Bayesian evidence combination (posterior odds = prior odds ×
likelihood ratio in log-odds space — not a weighted average dressed up as
"Bayesian"), explains its reasoning via SHAP/LIME so an analyst can see
*why* a score is high before acting on it, and only drafts an
incident-response playbook for events that clear a documented confidence
threshold (`final_risk_probability > 70`, enforced twice) — so the system
adds a filtered, correlated signal to the queue instead of one more raw
alert.

**What it is, concretely:** four integrated modules — statistical network
anomaly detection, phishing/content classification with Bayesian fusion, an
explainability dashboard, and an SOC orchestrator that gates LLM-drafted
playbooks behind that risk threshold. Built for SOC workflows and AI
hackathons alike.

## 🔍 On these metrics: what they do and don't prove

Two different questions get conflated when a project reports 0.999-style
AUC numbers, and it's worth separating them explicitly:

**1. "Is there data leakage / is the train-test split invalid?"** — No, this
part is verified. Both modules split before any fitting (stratified
train/test, and for Module 2 an additional strict text-deduplication check),
the decision threshold is swept on training data only, and — new in this
update — both modules now also report **5-fold cross-validation** (mean ±
std across folds, not one split) and a **separate stress-test holdout**
built from a harder slice of each module's own synthetic-data generator.
None of these change the conclusion: the numbers aren't a leakage artifact.

**2. "Will this perform this well on real-world traffic?"** — Unverified,
and the honest answer is *probably not, and Module 1's own stress test shows
why*. Every number in this repo, including the cross-validation and
stress-test numbers, comes from **synthetic, template-generated data**. That
data was deliberately made harder over several iterations (sophistication
tiers, hard negatives, geographic/time context), but "harder synthetic data
generated by the same process as the training data" is still not "traffic
from a real network or a real attacker's live campaign." Concretely:

- **Module 1** (network anomaly detection): the documented 0.81 AUC already
  reflects the current, harder data. But its own stress test — a harder
  holdout built from the same generator's stealthy-attack and noisy-normal
  parameters — pushes AUC to **0.18** (worse than random ranking). That's a
  genuine, reproducible finding: this specific unsupervised ensemble, on
  this 8-feature representation, can be fooled by low-and-slow attacks that
  resemble benign network noise. This is exactly the kind of failure mode a
  real deployment needs to know about *before* trusting the score, not after.
- **Module 2** (phishing detection) holds up on its own harder stress test
  (tier3-only "advanced" phishing), but that tier is still generated by the
  same templates the model trained on — lexical patterns a real, novel,
  human-written spear-phishing email might not share. A clean stress-test
  result here is evidence against *this specific kind* of overfitting, not
  evidence of real-world generalization.

**A realistic path to real-data validation** (not yet done, listed as future
work, not overclaimed as complete):
- Module 1: validate against a public labeled network-traffic dataset with
  real attack traffic — e.g. **CICIDS2017** or **UNSW-NB15** — re-deriving
  the risk-band thresholds and re-tuning the ensemble weights against that
  distribution rather than assuming the synthetic-data tuning transfers.
- Module 2: validate against a public phishing-email corpus (e.g. the
  **Nazario phishing corpus** or a similar public dataset) mixed with a
  sample of genuinely legitimate mail, to test generalization beyond this
  generator's template vocabulary.
- Both: if real labeled data isn't available yet, a useful interim step is
  adversarial red-teaming — have a person (not the data generator) write a
  handful of attack/phishing examples by hand, with no visibility into the
  generator's templates, and see where the models actually fail.

None of this is a reason not to use the architecture — the subprocess
isolation, the genuine Bayesian fusion, the explainability layer, and the
gated playbook generation are all sound engineering choices independent of
the specific numbers above. It's a reason not to read "0.999 AUC on
synthetic data" as "production-ready," which this README no longer implies
should be assumed.

## 🚀 Quick Start

### Prerequisites
- Python 3.8+
- pip

### One-Command Setup & Demo

```bash
# Clone and enter the repo
git clone https://github.com/bhaskar21-7/CyberShield-AI
cd CyberShield-AI

# Run the all-in-one demo (trains all modules, runs orchestrator, generates playbook)
bash demo.sh --mock-llm
```

This script will:
1. Train Module 1 (anomaly detection)
2. Train Module 2 (phishing classification)
3. Build Module 3's dashboard dataset
4. Launch Module 4's orchestrator on 10 sample events
5. Display threat scores, explanations, and auto-generated incident playbooks

No environment variables or manual file juggling required.

### Manual Setup (if you prefer step-by-step)

```bash
# 1. Install dependencies for each module
pip install -r module1/requirements.txt
pip install -r module2/requirements.txt
pip install -r module3/requirements.txt
pip install -r module4/requirements.txt

# 2. Train modules 1 & 2 (generates synthetic data if missing)
cd module1/src && python train.py && cd ../..
cd module2/src && python train.py && cd ../..

# 3. Build module 3's unified dataset + launch dashboard
cd module3/src && python build_dataset.py && cd ..
streamlit run app.py

# 4. Run the orchestrator (with LLM gating)
cd module4/src && python main.py --batch 10 --mock-llm
```

## 📊 Module Details

### Module 1: Statistical Anomaly Detection
Detects network traffic anomalies using an ensemble of **IsolationForest** (0.6 weight) + **OneClassSVM** (0.4 weight).

**Key Features:**
- 8 engineered features (connection density, burst score, traffic ratio, etc.)
- Outputs: `anomaly_score` (0-100), `risk_level` (Low/Medium/High)
- Measured 0.8086 AUC on held-out test split (25% of 32K synthetic rows); 0.8023 ± 0.0101 across 5-fold CV
- Permutation-importance feature ranking included

[→ Module 1 Full Details](./module1/README.md)

### Module 2: Phishing Detection + Bayesian Risk Fusion
Classifies malicious content across 5 channels (email, SMS, URL, login, API payload) using **TF-IDF + LightGBM**, then fuses with Module 1's network signal via **genuine Bayesian updating** (log-odds space, not weighted averaging).

**Key Features:**
- 0.9999 AUC vs. Naive Bayes (verified superior under strict deduplication)
- Real `shrinkage` calibration fix (documented overconfidence problem found & solved)
- Outputs: `phishing_probability` (0-1), `final_risk_probability` (0-1 posterior)
- Stress-tested against miscalibrated inputs

[→ Module 2 Full Details](./module2/README.md)

### Module 3: Explainable AI Dashboard
Interactive Streamlit dashboard explaining both Modules 1 & 2 via **SHAP** (global/local) and **LIME** (text-specific).

**Key Features:**
- LightGBM surrogate (fidelity: 99.6% vs. Module 1's real ensemble — measured, not claimed)
- 3 pages: Threat Overview | Explainability | Interactive Threat Explorer
- SHAP values verified for mathematical additivity property
- YARA/Snort/Sigma rule validation included

[→ Module 3 Full Details](./module3/README.md)

### Module 4: SOC Assistant Orchestrator
Wires all 3 modules into a single event pipeline, generates incident-response playbooks **only when risk > 70** (enforced 2× for safety).

**Key Features:**
- Deterministic templated rules (YARA/Snort/Sigma rules verified to compile/parse)
- LLM-narrated sections (threat summary, root cause, exec summary) + auto-generated commands/detections
- Curated threat intelligence (CVE/MITRE/category mapping — no hallucination)
- Append-only event log for auditability

[→ Module 4 Full Details](./module4/README.md)

## 🏗️ Architecture: Why It's Built This Way

### The Subprocess Isolation Pattern (Critical)
Each module has its own `utils.py` and Modules 1-2 both have `predict.py`. Python's import cache means importing multiple modules into the same process corrupts `sys.modules` for all of them — discovered and **fixed** during Module 2 development (AUC mysteriously dropped to 0.50 then bounced back after the fix was applied).

**Solution:** All cross-module calls go through **isolated subprocesses** (see `module_bridge.py` in each later module). Data flows via **temp JSON files**, not stdout (which carries logging noise) or CLI args (which hit OS length limits on large batches). Model artifacts (`.pkl` files) are safe to load directly via `joblib` since unpickling doesn't execute source code.

### Why Modules 1 & 2 Have Different Risk Bands
- Module 1: `Low (0-25), Medium (25-55), High (55-100)` — empirically derived from actual score distribution
- Module 2: `Low (0-40), Medium (40-70), Critical (70-100)` — fixed per spec
- They're not directly comparable — don't merge them without re-deriving.

### Why Module 3 Explains a Surrogate, Not Module 1's Real Ensemble
SHAP has no native support for OneClassSVM and partial/version-fragile support for IsolationForest. Rather than using the slow KernelExplainer, Module 3 trains its own fast LightGBM surrogate on Module 1's processed features, achieves 99.6% fidelity, and **reports that fidelity explicitly** on the dashboard (never presented as if it WERE Module 1's actual model).

## ❓ FAQ

**Q: Why are the module READMEs so long?**
A: They document every bug found and fixed, every design decision, and honest limitations. Judges appreciate thoroughness.

**Q: Can I use this in production?**
A: It's synthetic-data trained and hasn't been tested on real network traffic. Use the architecture and patterns as a foundation, but re-train on real data before deployment.

**Q: What if Module 4's LLM API is down?**
A: Use `--mock-llm` flag or set `SOC_LLM_MOCK_MODE=1`. The full orchestration pipeline still works — just without live narrative generation.

**Q: How do I know the SHAP explanations are correct?**
A: Module 3 verifies the SHAP additivity property numerically (base_value + sum(shap_values) == model's actual output for that row). Also cross-checks against permutation importance — independent methods landing on the same top features.
